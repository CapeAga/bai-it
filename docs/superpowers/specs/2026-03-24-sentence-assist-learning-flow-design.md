# 长难句低压力学习流设计

日期：2026-03-24

## 背景

`掰it` 现在已经具备两个关键能力：

- 自动把英文长难句掰开，降低结构理解难度
- 对选中的词提供英文解释、中文兜底和英式发音

下一步的问题不是继续堆功能，而是把长难句辅助设计成一条更符合学习目标的路径：

- 用户希望提升英文水平，而不是依赖中文翻译
- 中文仍然需要保留，作为看不懂时的安全网
- 体验必须轻，不要让阅读变成做题或上课

这个设计关注的不是“能提供多少信息”，而是“默认先提供哪一层信息”。

## 第一性原理

### 用户真正要完成的任务

用户在浏览英文网页时，核心任务不是学习语法，而是：

1. 先把当前内容看懂
2. 尽量维持阅读流，不要频繁被卡住
3. 在不增加明显负担的情况下，让英文能力缓慢增长

### 约束

- 阅读时的注意力极其稀缺
- 任何默认展开的重信息都会打断阅读节奏
- 一旦中文成为默认主通道，用户会迅速跳过英文理解
- 一旦产品变成“练习系统”，用户会因为压力而降低使用频率

### 推导出的产品原则

1. 默认帮助理解，不默认帮助翻译
2. 默认给用户最小足够信息，不一次性解释过多
3. 中文是兜底，不是主路径
4. 学习价值应该潜伏在阅读过程中，而不是额外制造任务
5. 所有交互都必须服务于“继续读下去”

## 设计目标

### 目标

- 让用户遇到长难句时更容易继续阅读
- 让用户优先通过英文理解英文
- 让用户每次卡住时都能顺手学到一点东西
- 保持低压力、低打扰

### 非目标

- 不把产品做成整页翻译器
- 不把长难句辅助做成语法课
- 不做默认评分、测验、连续打卡或待办队列
- 不为了“更强”而增加默认信息密度

## 推荐方案

### 总体思路

把长难句辅助固定成一条五层路径，从轻到重逐层展开：

1. 结构层：自动掰开句子，只展示结构
2. 简单英文层：对整句给一版更容易懂的英文改写
3. 中文兜底层：用户主动展开后才显示中文
4. 学习点层：只提炼一个最值得学的表达或结构提醒
5. 语感层：整句英式朗读

这条路径的目标不是让用户“学完一课”，而是让用户在一次卡顿中完成一次低负担的理解闭环。

## 方案比较

### 方案 A：中文优先

做法：
- 点击长难句后直接显示中文翻译
- 英文解释和表达点作为补充

优点：
- 理解最快
- 学习成本最低

缺点：
- 用户会默认跳过英文
- 很快形成对中文的依赖
- 与产品“学习而非翻译”的核心定位冲突

### 方案 B：学习优先

做法：
- 点击后默认展示句法讲解、表达分析、练习提示
- 中文弱化甚至隐藏

优点：
- 学习密度高
- 理论上更“有教育意义”

缺点：
- 认知负担过重
- 阅读场景不友好
- 很容易让用户觉得自己在上课

### 方案 C：理解优先，学习潜伏

做法：
- 默认只给最小必要辅助
- 先结构，再简单英文，再中文兜底
- 只保留一个学习点和一个朗读入口

优点：
- 最符合真实阅读场景
- 既能看懂，又能慢慢减少中文依赖
- 压力最低，最容易形成长期使用

缺点：
- 学习感不如重模式显性
- 对交互顺序要求更高

推荐：方案 C

## 交互设计

### 触发时机

触发对象：
- 已被识别的长难句
- 用户主动点按“掰it”的句子

不建议默认对整页所有句子同时展开更多解释，避免页面噪音。

### 默认状态

页面默认只显示：
- 掰开的句子结构
- 生词轻标记

默认不显示：
- 整句中文翻译
- 整句解释卡片
- 长篇语法分析

### 句子展开后的信息顺序

用户点击某个长难句后，句子下方或旁侧出现一个轻量句子卡片，顺序固定为：

1. `Simpler English`
   - 一版更简单、更短、更自然的英文改写
   - 目标是帮助用户继续用英文理解英文

2. `中文兜底`
   - 默认折叠
   - 用户点开才显示
   - 文案应强调“没看懂再看”，而不是鼓励优先点击

3. `这句最值得学的点`
   - 只允许一条
   - 形式可以是一个短语，或一个一句话结构提醒
   - 禁止多条并列讲解

4. `UK`
   - 朗读整句，而不是只读单词
   - 帮助用户把结构理解和语感联系起来

### 文案策略

文案应降低学习压迫感，不使用考试或训练语气。

建议：
- `Simpler English`
- `还没顺过来？展开中文`
- `这句最值得学的点`
- `听一遍 UK`

避免：
- `语法分析`
- `考考你`
- `你应该掌握`
- `本句知识点`

## 信息密度约束

### 必须收住的信息

句子卡片默认最多只展示：

- 1 条简单英文改写
- 1 条中文兜底
- 1 条学习点
- 1 个朗读按钮

### 不在句子卡片中做的事

- 不展示完整语法树
- 不展示多段表达总结
- 不展示 3 条以上例句
- 不展示正确率、完成度、练习进度
- 不展示“今日还剩几句没学”

## 压力控制机制

为了避免“适得其反”，这个功能必须有明确的减压设计：

1. 无强提醒
   - 不为句子解释做红点、徽标、待处理数

2. 无绩效反馈
   - 不评分
   - 不记录连续天数
   - 不展示“今天你还没学”

3. 可用即走
   - 用户看完一句就能继续读，不要求保存、标记、写笔记

4. 中文始终可达
   - 不能为了“沉浸”而把中文藏得太深
   - 用户卡住时必须能快速被接住

## 组件边界

### 页面内长难句卡片

职责：
- 承接单句理解
- 帮用户继续阅读

只负责：
- 简单英文
- 中文兜底
- 一个学习点
- 整句朗读

### 生词 tooltip

职责：
- 承接单词和短语理解

只负责：
- 单词或短语的简单英文解释
- 中文兜底
- 音标
- 英式发音

句子级能力不应塞回生词 tooltip。

### 管理页

职责：
- 阅读结束后的回看

不负责：
- 承担第一次理解

换句话说，句子第一次看懂这件事必须在页面内完成，而不是依赖用户跳到 Options 页。

## 数据与实现影响

### 数据需求

句子级解释结果建议扩展为：

- `simpler_english`
- `backup_chinese`
- `learning_point`
- `tts_text`（通常等于原句）

### 生成策略

- `simpler_english` 是句子级第一解释层，优先级最高
- `backup_chinese` 为短中文，避免长译文
- `learning_point` 严格限制为一条
- 结果应支持缓存，避免重复请求

### 无 API key 时

无 API key 不应假装提供完整句子解释。

合理行为：
- 保持已有本地掰句
- 允许整句本机英式朗读
- 引导用户配置 key 以获得 `Simpler English`

## 测试标准

以下情况通过才算设计落地成功：

1. 用户能在不看中文的情况下，比现在更容易理解长难句
2. 中文仍然能在 1 次操作内触达
3. 页面上不会同时出现过多解释层，视觉负担可控
4. 用户完成单句理解后可以立刻继续阅读
5. 功能不会向“翻译器”或“刷题器”方向漂移

## 建议的实现顺序

### P1

- 为长难句增加 `Simpler English`
- 中文改为折叠兜底
- 增加整句 `UK` 朗读

### P2

- 增加单条 `learning_point`
- 优化句子卡片的展开与收起动画

### P3

- 在管理页复用同样的信息层级
- 调整回味页，让其更贴近页面内的低压力逻辑

## 结论

对 `掰it` 这类产品，最优设计不是“解释更多”，而是“把默认解释层放在正确的位置”。

长难句辅助应当遵守：

- 先看结构
- 再看简单英文
- 还不懂再看中文
- 顺手学一个点
- 最后听一遍语感

这样既保护阅读流，也保护学习动机，最符合“功能在精不在多”的方向。
